# Как защищать проект

## Короткая формулировка проекта

Я сделал систему анализа новостей, которая:

1. собирает новости из внешнего источника;
2. сохраняет их в базу данных;
3. строит тематическую аналитику по корпусу;
4. показывает результаты в интерактивном интерфейсе;
5. позволяет искать похожие материалы и задавать вопросы по новостям.

## Какую проблему решает проект

Когда новостей много, человеку трудно:

- быстро понять, о чем в целом пишут СМИ;
- увидеть крупные темы;
- отследить, как меняются темы во времени;
- найти похожие материалы;
- быстро ответить на вопрос по большому массиву новостей.

Проект решает это через автоматический сбор, структурирование и базовую ML-аналитику.

## Почему выбрана такая архитектура

### 1. Парсер + БД + аналитика + интерфейс

Я разделил проект на четыре слоя:

- парсинг,
- хранение,
- аналитика,
- интерфейс.

Это удобно, потому что каждый слой можно менять отдельно. Например, можно заменить источник новостей, не переписывая интерфейс.

### 2. SQLite вместо PostgreSQL

Для учебного MVP я выбрал `SQLite`, потому что:

- она не требует отдельного сервера;
- быстро запускается;
- подходит для локального и облачного демо;
- хорошо работает на объеме учебного проекта.

То есть выбор был сделан в пользу простоты и скорости разработки.

### 3. Lenta.ru вместо RBC

Сначала рассматривался `RBC`, но у него были ограничения доступа и `403` на поисковый endpoint.

Поэтому я переключился на `Lenta.ru`, потому что:

- у нее более удобный поисковый endpoint;
- можно загружать большие выборки;
- это позволило быстрее получить рабочий корпус данных.

Это пример инженерного решения: не упираться в нестабильный источник, а выбрать более надежный для MVP.

## Какие шаги были реализованы

### Шаг 1. Сбор новостей

Что нужно было решить:

- получить новости за период;
- работать не с одной страницей, а с большим диапазоном дат;
- избежать ручной загрузки.

Что выбрано:

- парсер `Lenta.ru`;
- загрузка по временным чанкам;
- ограничение `size` и `chunk-days`, чтобы сайт не отдавал `504`.

Почему так:

- большие запросы к сайту вызывали timeout;
- разбиение периода на чанки сделало загрузку устойчивой.

### Шаг 2. Хранение данных

Что нужно было решить:

- где хранить статьи;
- как исключать дубли.

Что выбрано:

- `SQLite` как локальная БД;
- таблица `articles`;
- уникальность по `url`.

Почему так:

- URL естественно идентифицирует статью;
- SQLite достаточно для MVP и легко переносится.

### Шаг 3. Базовая аналитика

Что нужно было решить:

- как быстро понять, о чем корпус;
- как показать изменения во времени.

Что выбрано:

- частотные ключевые слова;
- динамика количества новостей по дням;
- кластеризация новостей.

Почему так:

- это дает быстрый и понятный результат;
- не требует сложного обучения моделей;
- хорошо подходит для учебной демонстрации.

### Шаг 4. Тематическая кластеризация

Что нужно было решить:

- выделить группы похожих новостей;
- не показывать просто “Topic 1”, а объяснять смысл темы.

Что выбрано:

- `TF-IDF + KMeans`;
- названия тем по ключевым словам кластера.

Почему так:

- `TF-IDF + KMeans` — простой и понятный базовый метод;
- хорошо объясняется на защите;
- легко показать, какие слова формируют тему.

### Шаг 5. Очистка текста

Что нужно было решить:

- в новостях встречаются слова, которые не несут смысл темы;
- они портят темы и similarity.

Что выбрано:

- расширенный список стоп-слов;
- добавлены доменные слова вроде `фото`, `репортаж`, `репортер`, `ТАСС`, `РИА`, `новости`, `источник`.

Почему так:

- без этого модель выделяла технические слова вместо смысловых;
- после очистки темы стали заметно лучше интерпретироваться.

### Шаг 6. Похожие статьи

Что нужно было решить:

- обычный поиск по словам находит не всегда действительно близкие материалы;
- хотелось показать элемент семантического поиска.

Что выбрано:

- `TF-IDF` по словам;
- символьные n-граммы;
- `cosine similarity`;
- небольшой бонус за совпадение темы.

Почему так:

- символьные n-граммы помогают лучше ловить похожие формулировки;
- бонус за тему делает similarity более устойчивым;
- это все еще легко объяснимо и быстро работает.

### Шаг 7. QA / локальный RAG

Что нужно было решить:

- пользователь должен иметь возможность задать вопрос по новостям;
- нужен простой RAG-подобный сценарий.

Что выбрано:

- локальный retrieval без внешней LLM;
- поиск наиболее релевантных новостей через `TF-IDF + cosine similarity`;
- выдача краткой выжимки по лучшим найденным материалам.

Почему так:

- это уже похоже на RAG по архитектурной идее;
- не требует отдельного API LLM;
- хорошо подходит для учебного проекта как первая версия.

## Какие фишки можно показать

На защите стоит сделать акцент на этих возможностях:

1. Автоматическая загрузка большого массива новостей.
2. Хранение корпуса в базе.
3. Тематическая кластеризация.
4. Осмысленные названия тем.
5. Динамика тем во времени.
6. Поиск похожих статей.
7. QA по текущему корпусу.
8. Готовый интерфейс, который можно открыть как приложение.

## Что говорить про ML-часть

Хорошая честная формулировка:

“В проекте использован не deep learning с обучением большой модели, а набор практических методов NLP и IR:

- TF-IDF,
- KMeans,
- cosine similarity,
- очистка текста,
- тематическая агрегация,
- retrieval по корпусу.

Это дало интерпретируемый и рабочий результат для учебного проекта.”

Такой ответ звучит честно и профессионально.

## Ограничения проекта

Важно прямо проговорить ограничения:

1. Источник пока один.
2. Темы строятся без ручной разметки.
3. Названия тем эвристические.
4. QA-блок retrieval-based, без полноценной генерации LLM.
5. Качество зависит от качества текстов и корпуса.

Это не минусы, а нормальные ограничения MVP.

## Что можно предложить как развитие

Если спросят “что дальше?”, можно ответить так:

1. Добавить несколько источников новостей.
2. Сохранять темы в отдельные таблицы БД.
3. Перейти на эмбеддинги вместо только TF-IDF.
4. Добавить полноценный RAG с LLM.
5. Улучшить нормализацию русского текста.
6. Добавить sentiment-анализ.
7. Построить дашборд по отдельным нарративам.

## Сценарий демонстрации

На защите можно идти так:

1. Показать главную страницу приложения.
2. Сказать, что данные автоматически собираются из новостей и лежат в БД.
3. Показать фильтрацию по датам.
4. Показать темы и их названия.
5. Показать динамику тем.
6. Открыть одну новость и показать похожие статьи.
7. Задать вопрос в QA-блоке.
8. Коротко объяснить, как это сделано технически.

Проект реализован как полноценное приложение с интерфейсом и подходит как MVP для анализа информационной повестки.”
